# 이벤트 백본 예제 — 트랜잭션·정합성 분석 (2026-07-09)

> **목적**: `examples/event-backbone-example`의 현재 트랜잭션 경계를 시나리오별로 정리하고, **동시성에서 상태가 깨지는 지점**을 짚어, 이 플랫폼이 원래 보장해야 할 **분산 정합성**으로 맞추는 방향을 제안. **분석·보고용(코드 작업 없음).**

---

## 0. 이 플랫폼이 지켜야 하는 것 (기준)

2PC(분산 트랜잭션) 없이도 **서비스 경계를 넘는 비즈 프로세스가 동시성·중복·순서 뒤섞임에도 일관·정확하게** 완결되는 것. 즉 사가가 "배차된 오더는 취소 불가" 같은 **불변식을 어떤 타이밍에도 지키는 것**이 핵심 가치다. (공통팀 in-house 역량 증명)

**트랜잭션의 기준(경계) = 한 서비스의 로컬 작업 1묶음.**
- 한 TX = (그 서비스 DB 도메인 변경) + (그 서비스 `outbox` 또는 `inbox` 기록). 서비스 안에서만 원자적.
- 서비스를 넘으면 = TX를 넘음 = **공유 롤백 없음.** 대신 at-least-once 메시징 + inbox 멱등 + 재시도 → **결국 일치(eventual consistency).**

---

## 1. 로컬 트랜잭션 카탈로그 (한 TX 안에 뭐가 묶이나)

| # | 서비스 | 트리거 | 한 TX 안의 쓰기 |
|---|---|---|---|
| create | OMS | HTTP `POST /orders` | orders(CREATED) + outbox(`OrderCreated`) |
| cancelOrder | OMS | HTTP `POST /orders/{id}/cancel` | (status 판정) CREATED→orders(CANCELLED)+outbox(`OrderCancelled`) / 그 외→outbox(`OrderCancelRejected`) |
| onMarkDispatched | OMS | consume `oms.cmd` | inbox + orders(DISPATCHED) |
| onMarkDelivered/Settled/Undispatched | OMS | consume `oms.cmd` | inbox + orders(해당 상태) |
| dispatch | TMS | HTTP `POST /dispatches` | trips(DISPATCHED) + outbox(`DispatchCreated`) |
| deliver | TMS | HTTP `.../deliver` | trips(DELIVERED) + outbox(`DispatchDelivered`) |
| cancelDispatch | TMS | HTTP `.../cancel` | trips(CANCELLED) + outbox(`DispatchCancelled`) |
| onCreateSettlement | BMS | consume `bms.cmd` | inbox + settlements(COMPLETED) + outbox(`SettlementCompleted`) |
| onOrderCreated | ORCH | consume `oms.order` | inbox + saga_instance(start CREATED) |
| onDispatchCreated | ORCH | consume `tms.dispatch` | inbox + saga(DISPATCHED) + outbox(`MarkDispatched`) |
| onDispatchDelivered | ORCH | consume `tms.dispatch` | inbox + saga(DELIVERED) + outbox(`MarkDelivered`+`CreateSettlement`) |
| onSettlementCompleted | ORCH | consume `bms.settlement` | inbox + saga(finish SETTLED) + outbox(`MarkSettled`) |
| onDispatchCancelled | ORCH | consume `tms.dispatch` | inbox + saga(CREATED 복귀) + outbox(`MarkUndispatched`) |
| onOrderCancelled | ORCH | consume `oms.order` | inbox + saga(finish CANCELLED) |

## 2. 시나리오별 트랜잭션 묶음

`~kafka~` = 서비스 경계(= TX 경계, 비동기). 각 줄 = 독립 로컬 TX.

**S1 정상 완주** (11 TX / 4 서비스)
```
TX1  [OMS ]  create           orders(CREATED)  + ob:OrderCreated
 ~kafka(oms.order)~
TX2  [ORCH]  onOrderCreated   inbox + saga(CREATED)
 (HTTP 배차)
TX3  [TMS ]  dispatch         trips(DISPATCHED) + ob:DispatchCreated        ★배차 확정
 ~kafka(tms.dispatch)~
TX4  [ORCH]  onDispatchCreated inbox + saga(DISPATCHED) + ob:MarkDispatched
 ~kafka(oms.cmd)~
TX5  [OMS ]  onMarkDispatched inbox + orders(DISPATCHED)                    ★오더 상태 동기화
 (HTTP 배송)
TX6  [TMS ]  deliver          trips(DELIVERED) + ob:DispatchDelivered
 ~kafka~
TX7  [ORCH]  onDispatchDelivered inbox + saga(DELIVERED) + ob:MarkDelivered + ob:CreateSettlement
 ~kafka~  TX8[OMS] onMarkDelivered → orders(DELIVERED)   /   TX9[BMS] onCreateSettlement → settlements(COMPLETED)+ob:SettlementCompleted
 ~kafka~
TX10 [ORCH]  onSettlementCompleted inbox + saga(SETTLED) + ob:MarkSettled
 ~kafka~
TX11 [OMS ]  onMarkSettled    inbox + orders(SETTLED)
```
→ **TX3(배차 확정)과 TX5(오더 상태변경)는 서로 다른 서비스·다른 TX.** TX5가 실패해도 TX3은 커밋된 채 남고, TX5는 재전달·재시도로 결국 성공(수렴). 이게 롤백 대신 재시도로 맞추는 정상 동작.

**S2 배차취소→미배차 복귀**: create → dispatch(→DISPATCHED) → cancelDispatch[TMS trips(CANCELLED)+ob:DispatchCancelled] → ORCH onDispatchCancelled[saga(CREATED)+ob:MarkUndispatched] → OMS onMarkUndispatched[orders(CREATED)].
**S3 재배차→완주**: S2 뒤 dispatch(새 trip) → S1 후반 재개.
**S4 미배차 취소**: create → cancelOrder[orders(CANCELLED)+ob:OrderCancelled] → ORCH onOrderCancelled[saga(CANCELLED)].
**S5 배차된 오더 취소 거부**: create → dispatch(→DISPATCHED) → cancelOrder[OMS status=DISPATCHED로 판정 → ob:OrderCancelRejected, orders 불변].
**S6 중복 멱등**: 같은 이벤트 재전달 → 소비측 inbox(ON CONFLICT)가 1회만.

## 3. 정합성이 깨지는 지점 (검증됨) ⚠️

**동시성 결함 — "배차 + 취소 동시"** (실측 3회 재현):
배차 직후(오더 status가 아직 CREATED로 보이는 창) 즉시 오더취소 →
```
최종:  OMS 오더 = DISPATCHED   /   saga = CANCELLED   (+ TMS trip = DISPATCHED)
```
**배차된 오더가 취소됐다가, 뒤늦은 `MarkDispatched`가 취소를 도로 DISPATCHED로 덮어씀.** 셋(오더·사가·배차)이 서로 다른 상태 = 정합성 붕괴. "배차된 오더 취소불가" 규칙이 창에서 무력화됨.

**근본 원인 4:**
1. **권한 분열** — 오더 생애주기 상태가 OMS(`order.status`)와 ORCH(`saga_instance`) **양쪽에** 존재. 진실이 둘.
2. **`Mark*` 무조건 덮어쓰기** — `onMarkDispatched`가 현재 상태 불문 DISPATCHED로 → CANCELLED를 덮음(전이 가드 없음).
3. **불변식을 '지연된 투영'으로 판정** — 취소 규칙을 OMS의 `order.status`(동기화 지연분)로 판정 → 창에서 우회됨.
4. **오더 단위 직렬화 부재** — 같은 오더 이벤트가 여러 토픽(oms.order/tms.dispatch/bms.settlement)에서 **동시 소비** → `saga_instance` 갱신 경합(락/버전 없음).

> S5(배차된 오더 취소 거부)는 **대기를 두면** 통과하지만, **동시/빠른 호출**에선 위처럼 뚫린다. 데모 타이밍 문제가 아니라 설계 결함.

## 4. 목표 상태 — 전이/불변식 표 (이렇게 지켜져야)

권위 상태(사가) 기준. `허용`=전이 실행 / `거부`=규칙상 불가(거부 이벤트) / `무시`=중복·이미 지난 것.

| 현재\액션 | 배차 | 배송완료 | 배차취소 | 오더취소 |
|---|---|---|---|---|
| CREATED | 허용→DISPATCHED | 거부(배차前) | 거부(배차없음) | **허용→CANCELLED** |
| DISPATCHED | 무시(중복) | 허용→DELIVERED | 허용→CREATED(복귀) | **거부(규칙)** |
| DELIVERED | 무시 | 무시(중복) | 거부(배송完) | 거부 |
| SETTLED | 무시 | 무시 | 무시 | 거부 |
| CANCELLED | 무시 | 무시 | 무시 | 무시(멱등) |

핵심: **오더취소 허용/거부 판정은 이 권위 상태(사가)로** 해야 창 없이 규칙이 지켜진다.

## 5. 수정 방향 (제안 — 결정 필요)

**골격(per-module API·프로세스 매니저)은 유지, 정합성 축만 리팩터:**

- **A. 오더 생애주기 단일 권한 = orchestrator 사가.** `saga_instance`가 유일한 진실. 모든 전이·불변식(위 §4 표)을 사가에서 권위 상태로 판정. → "배차된 오더 취소불가"도 사가가 판정(OMS 투영 아님).
- **B. OMS `order.status`는 순수 투영으로 격하.** 사가가 확정한 전이만 반영. 규칙 판정에 사용 안 함. `Mark*`는 **멱등·전진 전용**(CANCELLED를 덮지 않음 — 유효 전이만).
- **C. 오더 단위 직렬화.** `saga_instance`에 version 컬럼(낙관적 잠금) 또는 오더 단위 순차 처리 → 동시 이벤트 경합 차단. 충돌 시 재시도.
- **D. 취소 라우팅 재고.** 오더취소를 OMS가 자체 판정(현재) 대신, **사가가 권위 상태로 판정**하도록. (OMS는 취소 요청을 사가에 알리고, 사가가 허용 시 확정 커맨드.) — 이 지점이 §4 표를 강제하는 핵심.

**미결 결정 포인트:**
1. **불변식을 어디서?** 사가(권위) 단일 판정 [권고] vs OMS 애그리거트(투영 최신성 보장 필요).
2. **오더취소 경로**: OMS 직접 판정 유지 + 투영 최신성 보장(어려움) vs 사가 경유 판정(권고).
3. **직렬화 수단**: `saga_instance` 낙관적 잠금(version) vs 오더 단위 파티션/순차 소비.
4. **투영 정합**: OMS `order.status`를 없애고 사가만 볼지 vs 투영 유지(사가→OMS 단방향 sync).

> 확정 §7.1.7(종료상태 가드·멱등·정합성)과 정면으로 맞물리는 결정. 방향 정해지면 ① §4 표 확정 → ② 사가 단일 권한 구현 → ③ **동시성 시나리오(배차+취소 동시 등)로 검증**.
