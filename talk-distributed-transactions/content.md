## 슬라이드 2~4. 왜 분산 트랜잭션인가? (5분)

### 슬라이드 2 — 모놀리식에서는 쉬웠다

- 주문 생성 + 재고 감소 + 결제 처리 → 하나의 DB 트랜잭션
- `BEGIN` / `COMMIT` / `ROLLBACK` 으로 원자성 보장
- 실패하면 전부 없었던 일

### 슬라이드 3 — 마이크로서비스로 쪼개지면?

```
주문 서비스 (order-db)
재고 서비스 (inventory-db)
결제 서비스 (payment-db)
```

- 각 서비스는 자기 DB만 제어 가능
- 주문 성공 + 재고 성공 + **결제 실패** → 데이터 불일치
- 단일 트랜잭션으로 묶을 수 없다

### 슬라이드 4 — 분산 트랜잭션이 해결하려는 것

- 여러 서비스에 걸친 작업의 **원자성** 보장
- 완전히 성공하거나, 실패하면 전부 원래 상태로
- 오늘 다룰 방법: 2PC, 3PC, Saga, TCC

## 슬라이드 5~7. 2PC / 3PC (8분)

### 슬라이드 5 — 2PC 개요

```
Phase 1 (Prepare)
  Coordinator → Participant A: "커밋 가능?"
  Coordinator → Participant B: "커밋 가능?"
  Coordinator → Participant C: "커밋 가능?"

Phase 2 (Commit or Abort)
  모두 OK → Coordinator → 전체 COMMIT
  하나라도 NO → Coordinator → 전체 ROLLBACK
```

- DB 수준에서 원자성 보장
- XA 표준 (Java: JTA)

### 슬라이드 6 — 2PC의 한계

- **Blocking**: Coordinator가 죽으면 Participant들이 lock 상태로 무한 대기
- **SPOF**: Coordinator가 단일 장애점
- **성능**: 모든 참여자가 응답할 때까지 lock 보유 → 처리량 저하
- **네트워크 파티션**: Phase 2에서 일부만 COMMIT된 채 Coordinator 죽으면 불일치 발생 가능

### 슬라이드 7 — 3PC: 개선 시도와 그 한계

- Phase 추가: Prepare → **Pre-Commit** → Commit
- Pre-Commit으로 Coordinator 없이도 Participant가 판단 가능 → Non-blocking
- **그래도 네트워크 파티션에는 취약**
  - 파티션 상황에서 일부는 Commit, 일부는 Abort 가능
- 실제 운영 환경에서 거의 안 씀

## 슬라이드 8~11. Saga Pattern (8분)

### 슬라이드 8 — Saga의 핵심 아이디어

- 긴 트랜잭션을 짧은 **로컬 트랜잭션의 시퀀스**로 분해
- 각 로컬 트랜잭션은 성공 시 다음 단계 트리거
- 실패 시 **보상 트랜잭션(Compensating Transaction)** 으로 롤백
- 2PC처럼 lock을 오래 잡지 않음 → 높은 가용성

### 슬라이드 9 — Choreography Saga

```
주문생성 이벤트 발행
  → 재고 서비스가 구독, 재고 예약 후 이벤트 발행
  → 결제 서비스가 구독, 결제 처리 후 이벤트 발행
  → 실패 시 보상 이벤트 역순 발행
```

- 중앙 조율자 없음, 서비스끼리 이벤트로 통신
- 장점: 느슨한 결합
- 단점: 전체 흐름 파악 어려움, 디버깅 복잡

### 슬라이드 10 — Orchestration Saga

```
Orchestrator → 주문 생성    → 성공
Orchestrator → 재고 예약    → 성공
Orchestrator → 결제 처리    → 실패
Orchestrator → 재고 반납 (보상) → 완료
Orchestrator → 주문 취소 (보상) → 완료
```

- 중앙 오케스트레이터가 전체 흐름 제어
- 장점: 흐름이 한 곳에 모여 있어 파악·디버깅 용이
- 단점: 오케스트레이터가 너무 많은 것을 알게 됨

### 슬라이드 11 — Saga의 한계

- **최종 일관성(Eventual Consistency)**: 중간 상태에서 불일치 노출 가능
- **보상 트랜잭션의 복잡성**: 모든 스텝에 보상 로직 필요
- **멱등성 보장 필요**: 재시도 시 중복 실행 방지
- **격리성 없음**: 다른 트랜잭션이 중간 상태를 읽을 수 있음

## 슬라이드 12~13. TCC (5분)

### 슬라이드 12 — TCC 개요

```
Try     : 리소스 예약 (임시 점유)
Confirm : 예약 확정 (실제 반영)
Cancel  : 예약 취소 (임시 점유 해제)
```

- 2PC와 유사하지만 **비즈니스 레이어에서 제어**
- DB lock 대신 비즈니스 상태로 관리
- 예: 결제에서 "잔액 감소" 대신 "잔액 예약" → 나중에 확정/취소

### 슬라이드 13 — TCC의 특징과 한계

- 장점: DB에 종속되지 않음, 이기종 시스템 간 트랜잭션 가능
- 단점
  - Try/Confirm/Cancel 3개 API를 모든 서비스에 구현해야 함
  - 구현 복잡도 높음
  - Confirm 실패 시 재시도 로직 필요 (멱등성 필수)

## 슬라이드 14~16. 데모 (10분)

### 슬라이드 14 — 데모 시나리오 소개

```
주문 서비스 → 재고 서비스 → 결제 서비스

[정상] 전체 성공
[실패] 재고 없음 → 주문 취소
```

## 슬라이드 17~18. 한계 & 트레이드오프 정리 (5분)

### 슬라이드 17 — CAP Theorem과의 관계

```
C (Consistency)        — 일관성
A (Availability)       — 가용성
P (Partition Tolerance) — 네트워크 파티션 허용
```

- 분산 시스템에서 P는 포기 불가
- 2PC: CP 선택 → 가용성 희생
- Saga: AP 선택 → 강한 일관성 희생 (최종 일관성)

### 슬라이드 18 — 패턴 선택 기준

| 상황 | 추천 패턴 |
|---|---|
| 강한 일관성 필수, 짧은 트랜잭션 | 2PC |
| 긴 트랜잭션, 가용성 중요 | Saga (Orchestration) |
| 느슨한 결합, 이벤트 기반 아키텍처 | Saga (Choreography) |
| 이기종 시스템, DB 독립성 필요 | TCC |

> "은총알은 없다. 비즈니스 요구사항에 맞는 패턴을 선택하라."
