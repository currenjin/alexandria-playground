2PC

Two-Phase Commit

2PC

  • BlockingCoordinator 장애 → Participant 무한 대기

Prepare → Pre-Commit → Commit

Saga

Choreography각 서비스가 이벤트를 발행하고, 관심 있는 서비스가 구독한다

Saga

Choreography · 실패 흐름실패 이벤트를 앞 서비스들이 구독해 각자 롤백

Saga

Orchestration지휘자가 명령하고 응답을 기다린다

Saga

Orchestration · 실패 흐름실패 응답을 받으면 역순으로 보상 명령을 발행

Saga

한계.

  • 최종 일관성중간 상태에서 불일치 노출
  • 보상 로직모든 스텝에 Compensating 설계 필요
  • 멱등성재시도 시 중복 실행 방지 필수
  • 격리성 없음타 트랜잭션이 중간 상태를 읽음
TCC

Try · Confirm · Cancel

DB lock이 아닌 비즈니스 상태로 관리.
잔액을 깎지 않고, 먼저 예약한다.

TCC

트레이드오프.

  • +DB에 종속되지 않음 · 이기종 시스템 가능
  • −서비스마다 3개 API 구현 필요
  • −Confirm 실패 시 재시도 → 멱등성 필수

CAP

CAP 정리.분산 시스템은 세 가지 중 두 가지만 가질 수 있다

  • C · Consistency모든 노드가 같은 데이터를 본다
  • A · Availability요청은 언제나 응답을 받는다
  • P · Partition Tolerance노드 간 통신이 끊겨도 시스템은 계속 동작한다
