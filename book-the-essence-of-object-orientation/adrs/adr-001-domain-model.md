# ADR001. 도메인 모델에 책임 주도 설계 적용

## Context
현재 도메인 모델이 데이터 중심으로 설계되어 있어 다음과 같은 문제가 발생한다:
- 비즈니스 로직이 서비스 계층에 집중되어 있다 (빈약한 도메인 모델)
- 도메인 객체가 getter/setter만 가진 데이터 구조체에 가깝다
- 도메인 객체의 상태가 외부에서 직접 변경된다
- 변경 시 여러 서비스에 영향이 퍼진다

## Decision
책임 주도 설계(RDD)를 적용하여 도메인 모델을 리팩토링한다:

1. **행동 중심 설계**: getter/setter 대신 의미 있는 행동 메서드를 정의한다
   ```java
   // Before
   order.setStatus(OrderStatus.CANCELLED);

   // After
   order.cancel(cancellationReason);
   ```

2. **캡슐화 강화**: 상태 변경은 반드시 객체 내부 메서드를 통해서만 이루어진다

3. **협력 설계**: 유스케이스를 분석하여 객체 간 협력을 먼저 설계한다

4. **책임 할당**: Information Expert 원칙에 따라 정보를 가장 잘 아는 객체에게 책임을 할당한다

## Consequences

### Positive
- 도메인 지식이 도메인 객체에 캡슐화된다
- 변경의 영향 범위가 줄어든다
- 비즈니스 규칙의 응집도가 높아진다
- 테스트가 용이해진다

### Negative
- 초기 설계 비용이 증가한다
- 팀 전체의 DDD/RDD 이해가 필요하다
- 기존 코드와의 일관성이 깨질 수 있다

### Risks
- 점진적으로 적용하지 않으면 큰 리팩토링이 필요하다
- 팀 합의 없이 진행하면 코드 스타일이 혼재된다
