# Ch05. 책임과 메시지

## TL;DR
- **메시지**는 객체 간 유일한 의사소통 수단이다.
- 메시지가 객체를 결정하고, **메시지가 인터페이스를 결정**한다.
- **다형성**은 동일한 메시지에 대해 다른 방식으로 응답하는 능력이다.
- 인터페이스는 **최소 퍼블릭 인터페이스**를 유지해야 한다.

## Key Ideas
- **자율적인 책임**:
  - 객체가 "어떻게"가 아닌 "무엇을" 해야 하는지만 알려줘야 한다.
  - 책임이 자율적일수록 객체의 응집도가 높아지고 결합도가 낮아진다.
- **메시지(Message)**:
  - 수신자.메서드명(인자) 형태로 구성된다.
  - 메시지는 협력을 위한 유일한 수단이다.
  - 메시지를 먼저 결정하고, 그 메시지를 처리할 객체를 나중에 선택한다.
- **메서드(Method)**:
  - 메시지를 처리하는 구체적인 방법이다.
  - 같은 메시지라도 수신 객체에 따라 다른 메서드가 실행된다.
- **다형성(Polymorphism)**:
  - 동일한 메시지에 대해 서로 다른 객체가 다르게 응답하는 것이다.
  - 송신자는 수신자의 구체적인 타입을 몰라도 된다.
  - 대체 가능성을 통해 확장에 열려있고 수정에 닫혀있는 설계가 가능하다.
- **인터페이스**:
  - 퍼블릭 인터페이스: 외부에서 접근 가능한 메시지의 집합
  - 최소 인터페이스: 필요한 것만 노출한다.
  - 추상적 인터페이스: 구현 세부사항이 아닌 의도를 드러낸다.
- **인터페이스와 구현의 분리**:
  - 인터페이스는 "무엇"을, 구현은 "어떻게"를 담당한다.
  - 구현을 숨기면 변경의 영향을 최소화할 수 있다.

## Trade-offs

| 관점 | 장점 | 단점 | 언제 |
|------|------|------|------|
| 구체적 책임 | 명확하고 이해하기 쉽다 | 재사용이 어렵고 결합도가 높다 | 변경 가능성이 낮을 때 |
| 추상적 책임 | 유연하고 재사용이 가능하다 | 과도하면 이해가 어렵다 | 다양한 구현이 필요할 때 |
| 넓은 인터페이스 | 다양한 기능을 제공한다 | 결합도가 높고 변경이 어렵다 | 클라이언트 종류가 하나일 때 |
| 좁은 인터페이스 | 결합도가 낮고 변경이 쉽다 | 기능 접근이 제한적이다 | 대부분의 경우에 적합하다 |
